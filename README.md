# Система аутентификации и авторизации (RBAC)

Собственное backend-приложение, реализующее гибкую систему управления доступом на основе ролей (RBAC) с использованием **FastAPI**, **SQLAlchemy (async)** и **JWT**. Проект демонстрирует глубокое понимание процессов аутентификации, авторизации и управления правами без полного использования встроенных механизмов фреймворка.

## Технологии

- **Python 3.10+**
- **FastAPI** – веб-фреймворк
- **SQLAlchemy 2.0** (async) – ORM с паттерном Repository для пользователей
- **SQLite** (с `aiosqlite`) – база данных (легко заменяется на PostgreSQL)
- **Pydantic** – валидация данных
- **python-jose** – JWT (токены доступа)
- **passlib[bcrypt]** – хеширование паролей
- **uvicorn** – ASGI сервер

| `blacklisted_tokens` | Хранит JWT-токены, добавленные в чёрный список (для logout). |


## Система разграничения прав доступа

Реализована **собственная система RBAC**, работающая на уровне зависимостей FastAPI.

### Принцип работы

1. **Аутентификация** – проверяется JWT-токен, извлекается пользователь через `get_current_user`.
2. **Авторизация** – проверяется наличие у пользователя необходимого разрешения через его роли.
   - Если у пользователя есть роль `admin`, доступ разрешается автоматически.
   - Иначе ищется разрешение с заданными `resource` и `action` среди всех разрешений всех ролей пользователя.
3. **Обработка ошибок**:
   - `401 Unauthorized` – не передан токен, токен недействителен или истёк, либо пользователь заблокирован (`is_active=False`).
   - `403 Forbidden` – пользователь аутентифицирован, но не имеет прав на запрашиваемый ресурс.

### Управление правилами

- **Администратор** (роль `admin`) имеет полный доступ к API управления ролями и разрешениями.
- Через эндпоинты `/admin/roles` и `/admin/permissions` можно создавать, редактировать, удалять роли и разрешения, а также назначать/отзывать разрешения у ролей.
- Назначение ролей пользователям выполняется через `/admin/users/{user_id}/roles`.

### Используемые зависимости

- `get_current_user` – извлекает текущего пользователя из токена, проверяет активность и наличие в чёрном списке.
- `get_current_active_user` – гарантирует, что пользователь активен.
- `require_admin` – требует наличия роли `admin`.
- `require_permission(resource, action)` – фабрика, возвращающая зависимость, проверяющую конкретное разрешение.

## Установка и запуск

### 1. Клонирование репозитория

```bash
git clone git@github.com:Popug1ch/Auth.git
```

### 2. Создание виртуального окружения

```bash
python -m venv venv
source venv/bin/activate      # Linux/Mac
venv\Scripts\activate         # Windows
```

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
```

### 4. Настройка переменных окружения

Создайте файл `.env` в корне проекта (опционально). По умолчанию используются значения из `config.py`:

```
SECRET_KEY=your-secret-key
ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=30
```

### 5. Инициализация базы данных и тестовых данных

При первом запуске приложение автоматически создаст таблицы и наполнит их тестовыми данными (см. раздел [Тестовые данные](#тестовые-данные)).

### 6. Запуск сервера

```bash
python main.py
```

Сервер будет доступен по адресу: `http://127.0.0.1:8000`

**Документация API** (Swagger UI) доступна по адресу:  
`http://127.0.0.1:8000/docs`

### Роли
- **admin** – администратор (обладает всеми разрешениями)

- `user:read`, `user:update`, `user:delete`

- `resource:read`

|----------------------|-----------|--------|
| `admin@example.com`  | `admin123`| admin  |
| `user@example.com`   | `user123` | user   |

## API эндпоинты

Все эндпоинты возвращают JSON. Для защищённых запросов необходимо передавать JWT-токен в заголовке `Authorization: Bearer <token>`.

### Аутентификация (`/auth`)

| Метод | Эндпоинт            | Описание |
|-------|---------------------|----------|
| POST  | `/auth/register`    | Регистрация нового пользователя. Принимает `UserCreate` (email, пароль, подтверждение, ФИО). Возвращает `access_token`. |
| POST  | `/auth/login`       | Вход по email и паролю (тело `UserLogin`). Возвращает `access_token`. |
| POST  | `/auth/token`       | Вход для Swagger UI (использует `OAuth2PasswordRequestForm`). Возвращает `access_token`. |
| POST  | `/auth/logout`      | Выход из системы. Требует аутентификации. Добавляет текущий токен в чёрный список. |

> **Примечание:** `/auth/token` используется исключительно для интеграции с документацией Swagger. Для обычного входа используйте `/auth/login`.

### Пользователи (`/users`)

| Метод | Эндпоинт              | Описание |
|-------|-----------------------|----------|
| GET   | `/users/me`           | Получить информацию о текущем пользователе (включая роли и разрешения). |
| PUT   | `/users/me`           | Обновить данные профиля (email, ФИО, пароль). Все поля опциональны. |
| DELETE| `/users/me`           | Мягкое удаление аккаунта (устанавливает `is_active=False`). Дополнительное подтверждение не требуется (упрощённая реализация). |

### Администрирование (`/admin`) – только для роли `admin`

#### Пользователи (список)
| Метод | Эндпоинт            | Описание |
|-------|---------------------|----------|
| GET   | `/admin/users`      | Получить список всех пользователей с их ролями и разрешениями. |

#### Роли

| Метод | Эндпоинт                | Описание |
|-------|-------------------------|----------|
| GET   | `/admin/roles`          | Список всех ролей с их разрешениями. |
| POST  | `/admin/roles`          | Создать новую роль (тело `RoleCreate`). |
| PUT   | `/admin/roles/{id}`     | Обновить роль (тело `RoleUpdate`). |
| DELETE| `/admin/roles/{id}`     | Удалить роль (запрещено для системных ролей `admin` и `user`). |

#### Разрешения

| Метод | Эндпоинт                    | Описание |
|-------|-----------------------------|----------|
| GET   | `/admin/permissions`        | Список всех разрешений. |
| POST  | `/admin/permissions`        | Создать новое разрешение (тело `PermissionCreate`). |
| PUT   | `/admin/permissions/{id}`   | Обновить разрешение. |
| DELETE| `/admin/permissions/{id}`   | Удалить разрешение. |

#### Назначение ролей пользователю

| Метод | Эндпоинт                               | Описание |
|-------|----------------------------------------|----------|
| POST  | `/admin/users/{user_id}/roles`         | Назначить роль пользователю (тело `AssignRole`). |
| DELETE| `/admin/users/{user_id}/roles/{role_id}` | Удалить роль у пользователя. |

### Ресурсы (Mock) – `/resources`

Служат для демонстрации работы системы доступа.

| Метод | Эндпоинт            | Описание |
|-------|---------------------|----------|
| GET   | `/resources/`       | Доступен только с разрешением `resource:read`. Возвращает фиктивный список объектов (Project Alpha, Beta, Gamma). |

## Коды ошибок

- `401 Unauthorized` – отсутствует или недействителен токен, пользователь не активен или неверные учётные данные.
- `403 Forbidden` – токен действителен, но недостаточно прав для доступа к ресурсу (или попытка удалить системную роль).
- `400 Bad Request` – ошибки валидации (некорректные данные, пароли не совпадают, дублирование email/роли/разрешения и т.п.).
- `404 Not Found` – запись не найдена (например, роль или пользователь).
- `500 Internal Server Error` – ошибка сервера (например, отсутствует роль `user` при регистрации).

## Примеры запросов

### Регистрация

```http
POST /auth/register
Content-Type: application/json

{
  "email": "newuser@example.com",
  "password": "securepass",
  "password_confirm": "securepass",
  "first_name": "Иван",
  "last_name": "Петров",
  "father_name": "Сергеевич"
}
```

Ответ (токен + тип):
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "bearer"
}
```

### Логин (через /auth/login)

```http
POST /auth/login
Content-Type: application/json

{
  "email": "admin@example.com",
  "password": "admin123"
}
```

Ответ аналогичный – `access_token`.

### Логин через Swagger (/auth/token)

Используется форма OAuth2: `grant_type=password&username=admin@example.com&password=admin123`.

### Получение профиля

```http
GET /users/me
Authorization: Bearer <access_token>
```

Ответ (с ролями и разрешениями):
```json
{
  "id": 1,
  "email": "admin@example.com",
  "first_name": "Admin",
  "last_name": "User",
  "father_name": null,
  "is_active": true,
  "created_at": "2026-06-28T12:00:00+00:00",
  "updated_at": null,
  "roles": [
    {
      "id": 1,
      "name": "admin",
      "description": "Administrator",
      "permissions": [
        {"id": 1, "resource": "user", "action": "read", "name": "user:read", "description": null},
        ...
      ]
    }
  ]
}
```

### Обновление профиля

```http
PUT /users/me
Authorization: Bearer <token>
Content-Type: application/json

{
  "first_name": "НовоеИмя"
}
```

### Доступ к защищённому ресурсу (Mock)

```http
GET /resources/
Authorization: Bearer <access_token>
```

**Успех (200):**
```json
[
  {"id": 1, "name": "Project Alpha"},
  {"id": 2, "name": "Project Beta"},
  {"id": 3, "name": "Project Gamma"}
]
```

**Ошибка (403)**, если у пользователя нет разрешения `resource:read`:
```json
{
  "detail": "Permission denied"
}
```

### Администрирование: создание роли

```http
POST /admin/roles
Authorization: Bearer <admin_token>
Content-Type: application/json

{
  "name": "manager",
  "description": "Менеджер проектов",
  "permission_ids": [1, 12]
}
```

### Назначение роли пользователю

```http
POST /admin/users/3/roles
Authorization: Bearer <admin_token>
Content-Type: application/json

{
  "role_id": 4
}
```

## Дополнительные замечания

- **Чёрный список токенов** – при выходе из системы токен добавляется в таблицу `blacklisted_tokens`. При последующих запросах он будет отклонён даже до истечения срока действия.
- **Мягкое удаление** – пользователь с `is_active=False` не может войти в систему, его данные сохраняются в БД.
- **Тестирование** – рекомендуется использовать встроенную документацию Swagger (`/docs`) или инструменты типа Postman.
- **Замена БД** – для перехода на PostgreSQL достаточно изменить `DATABASE_URL` в `database.py` на `postgresql+asyncpg://...`.
- **Архитектура** – код разделён на слои: роутеры, зависимости, репозиторий, модели, схемы. Это облегчает масштабирование и поддержку.
